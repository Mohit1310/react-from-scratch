What is '.parcel-cache'?

It is a cache folder created by Parcel bundler. When we run parcel it budles all the code and the assets to improve the code performance. These all are stored in .parcel-cache folder. When you run the project first time it is created and next time when you run the project again it wil run faster than last time because everything is stored in cache folder.

.parcel-cache is useful because

  1. Faster builds
  2. Reduced Resource usage
  3. Efficient Development Workflow

What is '.gitignore'? What should we add and not add into it

It is a file in which we can specify the files that we don't want to see in our repo but can be viewed on our local machine.

- What to add?
  1. Generated files
  2. Dependencies
  3. Temporary files
  4. User-specific files
  5. Sensitive information (.env, etc)
- What not to add?
  1. Source code
  2. Configuration files
  3. Documentation
  4. Assets and resources
  5. Files required for testing

In general, the goal of .gitignore is to keep your repository clean, focused, and free from unnecessary clutter. By excluding files that are not crucial for the development process, you can maintain a more organized and efficient version control system.

What is the difference between package.json and package-lock.json

File Purpose Usage
package.json Dependency Declaration Specifies the project's dependencies, scripts, metadata, and configuration.
package-lock.json Dependency Locking Locks dependency versions and resolves them to ensure consistent installs.

In summary, while both package.json and package-lock.json are integral parts of Node.js projects managed with npm, they serve different purposes. The package.json file defines project metadata, scripts, and dependencies, while the package-lock.json file ensures that dependency versions are locked and consistent across different installations.

Why should I not modify package-lock.json?

Following are the reasons for not to modify the file

  • Dependency consistency
  • Automatic generation
  • Version conflicts
  • Collaboration Issues
  • Upgrades and installs
  • Potential Data loss

What is browserlists?

It is a configuration tool and library used to specify which browsers and their versions your web application should support. Browserslist is commonly used in web development, especially when setting up autoprefixing for CSS and making decisions about which browser versions to target for compatibility.
Browserslist allows you to define a list of browsers that your project needs to support. This list can be used by various tools and libraries to determine how to handle CSS and JavaScript transformations to ensure compatibility with the specified browsers.

Here's an example of how Browserslist is typically used:

  • In package.json
  • "browserslist": "> 1%, last 2 versions"
